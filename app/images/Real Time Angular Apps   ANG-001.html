Real Time Angular Apps

Estimated Time: 2-3 hours

The defining feature of a real time web app is that the contents of a page can update as soon as data on the backend changes. Whereas the Angular apps you've encountered so far have relied on AJAX calls to the backend server, for real time apps (RTAs), you take advantage of WebSockets to maintain a persistent, two-way connection to the backend. While this sounds simple enough, the devil is in the details.

Some practical use cases for RTAs are:

  • Live chat
  • Financial live updates
  • Multiplayer gaming
  • Social media feeds
  • Live analytical data
  • Alerts and notifications

In this assignment, you'll first consider how Angular can interact with WebSockets, and then you'll go on to look at some existing tools you can use to build a real time Angular app. Finally you will look at a simple Firebase integration in more detail as well as Google+ authentication and security.

However we will not be discussing the nature of reality or time in this assignment.

WebSockets + AngularJS

A standard Angular app uses AJAX (via $http) to download new data from the backend which is then placed in the scope. This typically means that all the data, based on the state of the application or the URL within the browser, is downloaded in one go and then used within the application scope.

In RTAs, the front-end code is expected to keep a persistent connection to the backend and relay updates in code to the appropriate parts of the application. In other words, when the ResourceA updates on the server (something like a list of comments for example) then the next thing to do is to update the binding on the front-end with the new updates of ResourceA so that the application can render the new data. So, in a nut-shell, all RTAs do is just synchronize themselves with the backend using WebSockets. The major difference, however, is that Web Sockets are designed to transport small amounts of data.

Therefore in order to get a broad understanding of whats out there, lets examine some existing tools you can use to take the difficulty out of building real time Angular apps.

There are a number of ways to achieve real time synchronization and all of which will require remote data persistence. Two of the more simple ways to implement RTAs on the web stack are: Pusher.js which does not persist for you and Firebase which can easily handle all of your data persistence with their provided back-end.

Firebase (AngularFire)

Firebase is a commercial backend-as-a-service platform for building real-time applications. AngularFire is a plugin maintained by Firebase that integrates Angular and Firebase. Firebase's approach to storing and relaying real-time data is to persist data on their own backend and then have your front-end Angular code connect to that backend using WebSockets. This means that you dont necessarily need a backend at all or, at the very least, you dont need to connect your backend to Firebase.

We'll walk you through some of the pieces of Angular Fire later in this assignment.

Pusher.js

Pusher.js is essentially just WebSockets + a transport channel. Data gets placed in a pusher channel and then relayed to all connected channels before being erased from the servers memory. This means that, unlike Firebase, none of the data is persisted on Pusher's server.

Since Pusher doesnt persist the data, it is up to you to configure your backend code to relay data to Pusher whenever any of the model data is updated. Once relayed, Pusher will automatically push it out to any connections that are connected to it via WebSockets (or any other socket-based system). In short, Pusher is just an easy to use notification platform.

Keep in mind that, since data transported through Pusher is entirely managed by the developer, the front-end code will have to listen on its own channels to detect when new data is coming in from the backend. This approach can make an application become complicated if done incorrectly.

Here are some resources to get you started:

  • Core Angular contributor Matias Niemela's presentation on Pusher and Angular.

  • Pusher blog post on integrating Angular, Pusher, and Node

Setting up your own Web Sockets

If you have the time and skill to configure your backend to relay data using its own WebSockets then the end result is similar to using Pusher. There is also Pubnub.js which is very similar to PusherJS; both of which have existing Angular integrations. Some alternatives to using Pusher are Socket.IO and SockJS ; both of which have numerous back-end implamentations which you will also need to setup. However, the major difference is that you are not paying for a 3rd party service.

That being said, builiding your own implementation of Pusher from scratch is probably not worth the time and maintenance investment. Unless you have a specific reason to not use an existing service like Pusher (for instance, handling sensitive information that you don't want to expose to a third party service),

Building your own implementation of Web Sockets can potentially expose security risks and scaling issues as well as number of event complications. So, it's recommended to use an off the shelf solution like Firebase or Pusher for building real time Angular apps.

Working with Firebase

Firebase is a free service that lets you save your data on a remote server that can then be shared between various users of your App and potentially other SPAs you may want to access this data. It can support and facilitate user logins, data security and real time data messaging.

To set up firebase you need to install the appropriate libraries into your Angular application as well as create a remote data store. The said store will be made available to use via a Firebase URL which you will connect to in your applications.

Create a Firebase 'database' and URL

  1. Create an account at https://www.firebase.com/signup/
  2. Copy the [domain].firebaseIO.com, this is Firebase App URL
  3. Alternatively, you can also create a new Firebase App to get another free URL

Some Help from AngularFire

The standard method of integrating Firebase into a JavaScript SPA involves managing events manually and to achieve this in Angular takes managing both connections and $watches. So you can instead use the AngularFirebase service.

At some point in your application setup cycle, run the following bower command:

bower install angularfire -S
Unless you wish to use the CDN service for angular firebase, in which case you may skip the above step and use the URLs provided at their Firebase API documentation .

This will download all the relevant files into your application. The -S flag saves the install dependencies to any bower.json files you have for the project. Then add the following script files to your HTML.

<script src="[vendor]/angular/angular.js"></script>
<script src="[vendor]/firebase/firebase.js"></script>
<script src="[vendor]/angularfire/dist/angularfire.js"></script>
Be sure to update the above links to point to your bower installation folder.

This will install everything you need to use the pre-made $firebase Angular directive. The said directive can then provide automatic sync between Angular's $scope model bindings and the Firebase URL you have created.

You will need to add the dependency for your module to the 'firebase' service:

var MyModule = angular.module('MyModule', [ 'firebase' ]);

This will now make the $firebase service available for injection after the 'config' stage has completed. This service takes a Firebase reference object that is tied to a URL. You= can create Firebase reference objects using the global Firebase constructor function. Let's do just that; you will put this Firebase reference object inside a simple factory that can be used throughout your module.

MyModule
    // change this to your Firebase URL
    .constant('FIREBASE_URL', '...firebaseIO.com')
    .factory('firebaseReference', function(FIREBASE_URL){
        return new Firebase( FIREBASE_URL );
    })

Now you can use the 'firebaseReference' service to connect to your Firebase endpoint and persist your data at Firebase! To use your new reference you can pass it to the $firebase service. Here's the full API documentation for the $firebase service; Firebase API documentation .

The following code passes the firebaseReference to the $firebase service and then saves it to your controller scope.

MyModule.controller('FireCtrl', 
function($scope, $firebase, firebaseReference){
    $scope.myData = $firebase(firebaseReference).$asObject();
})

Here you pass your reference to the $firebase service and then use the new and improved $firebase API method $asObject to convert the firebase data into a Firebase Object. Now, set a quick and dirty binding in your HTML to take a look. You'll see that right now it will contain no data:

<pre ng-controller="FireCtrl">
    {{ myData | json }}
</pre>

Saving Data to Firebase

You can access your Firebase forge by opening the Firebase URL you created, in a browser, in a new tab and you will see the endpoint's Dashboard. Here you can CRUD your data manually, quickly add a row by pressing the green plus symbol inside the Web interface and give your data a name and value. If you navigate back to your Angular application, you will see that the myData will have updated to your new data without you needing to refresh the page! Just like magic! This was pushed from Firebase to the application and the $firebase service has already handled this event for us.

Alternatively you can set an Array of data by writing an array literal in the value of the forge's data and then using the $asArray method instead of $asObject in your code. But clearly, you don't want to write into your datastore manually every time! And yet again the $firebase service can help you here greatly!

A Chat Directive

Here you make use of the child method on the Firebase reference to create a value named 'messages'. The child method will return a new firebase reference object that you then pass into $firebase.

Here's a quick chat directive example:

The directive's template HTML is further below.
angular.module('MyModule', [ 'firebase' ])
    // change this to your Firebase URL
    .constant('FIREBASE_URL', '...firebaseIO.com')
    .factory('firebaseRef', function( FIREBASE_URL ) {
        return new Firebase( FIREBASE_URL );
    })
    .directive('chat', function( ) {
        'use strict';
        return {
            restrict: 'E',
            templateUrl: "chat.tmpl.html",
            scope: {},
            controllerAs: 'ctrl',
            controller: function($scope, $firebase, firebaseRef){
                $scope.messages = $firebase(
                    firebaseRef.child('messages')
                ).$asArray();
                this.sendMessage = function(send){
                    // do some validation
                    if ( !send.message ) return false;
                    // save the message
                    $scope.messages.$add({
                        message: send.message.trim(),
                        userName: send.userName,
                        datetime: Date.now()
                    });
                    $scope.send.message = "";
                }
            }
        };
    })

In this directive you use new Firebase( FIREBASE_URL ) to create a reference object; to the firebase (database) that you can view at https://<domain>.firebaseio.com. You then use firebaseRef.child('messages') inside your controller to create a new 'child' (value) in the Firebase endpoint with a name of 'messages'. This could be considered similar to a database table with the name of 'messages'. So, now you're using a firebase reference object that points to the 'messages' value instead of your Firebase root value!

$firebase creates a connection to the child reference of 'messages' and $asArray() returns a new binding on the scope. In other words, when you add data to the firebase backend, it will also update $scope.messages. Firebase will be pushing these changes to the reference object and the $scope binding will be automatically updated, then the view simply iterates over these messages to display them as they change.

Many of the $firebase methods return a promise which you can also hook into with methods such as $loaded, as described in this article released by the firebase team.

NOTE: The AngularFire library was recently updated so many of the existing resources discussing AngularFire are no longer relevant; the article linked to in the above text is regarding the latest release.

Finally the sendMessage method will add a new message to your endpoint via the $add method which persists to the firebase store!

Here's the HTML for chat.tmpl.html so you can quickly set it up:

<div id="messages">
    <div class="message" ng-repeat="m in messages">
        <strong>{{ m.userName }}</strong> 
        <em>{{ m.datetime | date }}</em> : 
        <span>{{ m.message }}</span>
    </div>
</div>
<form name="chatForm" ng-submit="ctrl.sendMessage(send)">
    <fieldset>
        <input required name="userName" type="text"
            ng-model="send.userName" 
            id="userName" placeholder="Name">
        <input required name="message" type="text" 
            ng-model="send.message" 
            id="message" placeholder="Message&hellip;">
        <!-- submit button: pressing return submits -->
        <input hidden type="submit">
    </fieldset>
</form>

Open up your App with a different session (a new user in chrome or a different browser completely) and talk to yourself watching the data sync between both sessions via the Firebase endpoint!

It is also possible to skip the $add step described above (also $set, $remove, etc) by creating a 3-way-data-binding with the $bindTo method. You can create scope data that only needs be manipulated directly and it's changes are persisted to the backend.

Security

When looking at your Firebase URL in the browser, this is called the forge and you are provided with a number of configuration settings and options. One of the advanced features here is to make use of the security capabilities of Firebase. You can disable writing of data by changing or removing ".write" from your rules so that it evaluates to false. This is a bespoke security language created by the Firebase team that allows very complex queries based on user's sessions, data validation, times and more. You can find more information on this at the Firebase security guide .

You won't be going into much detail here but it's worth knowing that you use much JavaScript syntax inside strings and many keys are generated for you, such as the $valueid for any children in "value".

One common practice may be to set write access to authenticated users only. In the chat directive described above, you could make your messages object only writable by those who have logged in by using the settings below:

{
    "rules": {
        ".read": true,
        ".write": true,
        "messages" : {
            ".write": "auth != null",
        }
    }
}

This makes use of the auth variable that is provided by the security rules to enforce that the messages value can only be written to by users who are currently logged in!... but how do they login?

Google Authentication

You can use Firebase to login your application through many services such as Google+, Facebook, Twitter and even creating Email subscription lists.

<script src="[vendor]/firebase-simple-login/firebase-simple-login.js"></script>

To use it you must include the above file, which will provide another angular directive. This directive can then easily be used to create a login for your application. Much like the $firebase service, the $firebaseSimpleLogin requires a Firebase reference object to connect to an endpoint.

You will need to go into your forge and open the Simple Login area where you can enable various authentication endpoints for your application. The Firebase website has useful instructions on how to setup Google as a back-end using their Website Interface.

$firebaseSimpleLogin(ref).$login( 'google', {
    rememberMe: true,
    // describe your intentions
    scope: 'https://www.googleapis.com/auth/plus.login'
} );

Executing the above code will open a new browser window popup for logging into your application via existing Google+ accounts. It can easily be changed for a number of other methods of login. Once logged in the following event will fire:

$rootScope.$on('$firebaseSimpleLogin:login',

Here you can receive the user's successful login attempt and respond appropriately. Here is a more thorough example using pseudo code:

scope.user = User.get();
$rootScope.$on('$firebaseSimpleLogin:login',
  function (event, user) {
    if (user != null)
        User.setUser(user.thirdPartyUserData);
});
$rootScope.$on('$firebaseSimpleLogin:logout',
  function (event, user) {
    User.setDefault();
});

Using other features of $firebase such as $watch events and $extendFactory, you could persist logged in users to an array, combined with your security rules to make a chat room.
